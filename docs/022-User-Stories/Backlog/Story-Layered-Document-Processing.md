# User Story: Tách lớp xử lý tài liệu (enabler)

**As a** Đội phát triển DocsViewer,
**I want to** tách lớp logic `parse/extract` (Document Processing) độc lập với UI và tầng người dùng,
**So that** về sau có thể thêm định dạng mới hoặc gắn auth/multi-tenant mà không phải viết lại core.

> Đây là **enabler story** (architectural). Giá trị là khả năng mở rộng, không phải tính năng nhìn thấy trực tiếp.

## Acceptance Criteria

```gherkin
Scenario: Logic parse/extract tách rời khỏi UI
  Given kiến trúc DocsViewer
  When module xử lý tài liệu được gọi để parse/extract một file
  Then logic parse/extract không phụ thuộc trực tiếp vào component UI nào (KR3.1)
  And UI chỉ tiêu thụ kết quả qua một interface định nghĩa rõ ràng

Scenario: Extension point cho định dạng mới
  Given lớp Document Processing đã tách
  When cần hỗ trợ một định dạng mới (ví dụ .pptx ở M4)
  Then có extension point cho phép cắm processor mới mà không sửa core viewer (KR3.2)

Scenario: Extension point cho auth/multi-tenant
  Given lớp dữ liệu người dùng được tách khỏi lớp xử lý
  When chuẩn bị gắn auth/multi-tenant ở M3
  Then tồn tại điểm móc (extension point) rõ ràng để gắn vào mà không viết lại core (KR3.3)
```

## Unhappy Path

```gherkin
Scenario: File không parse được được bàn giao cho lớp xử lý
  Given lớp Document Processing nhận một file hỏng/không parse được
  When processor tương ứng cố xử lý file
  Then lớp xử lý trả về trạng thái lỗi tường minh cho UI
  And không đẩy nội dung sai/rỗng vào pipeline View/Extract

Scenario: Định dạng chưa có processor
  Given một file lọt vào lớp xử lý nhưng chưa có processor đăng ký cho định dạng đó
  When lớp xử lý tra cứu processor phù hợp
  Then hệ thống trả về lỗi "định dạng không được hỗ trợ" thay vì xử lý sai
```
