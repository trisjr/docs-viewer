# 🧠 Kinh Nghiệm Thiết Kế Workflow + Skill Architecture

## 📌 1. Tóm tắt (Summary)
Đúc kết từ phiên thiết kế workflow `/opsx-ingest` — một workflow phức tạp kết hợp nhiều capability (URL parsing, AI triage, document generation, bug report, impact analysis). Bài học chính: khi nào nên tách thành Skill riêng, cách tối ưu token budget, và các pitfall khi viết workflow instruction.

## 🧩 2. Mẫu hình & Giải pháp (Patterns & Solutions)

### A. Quy tắc Tách Workflow vs Skill
- **Tên mẫu hình:** Selective Decomposition (Phân tách Có Chọn lọc)
- **Bối cảnh áp dụng:** Khi một workflow có >3 khối chức năng rời rạc
- **Cách thực hiện:**
  1. Liệt kê tất cả khối chức năng trong workflow
  2. Đánh giá **tính tái sử dụng** (Reusability) của từng khối — "Khối này có được dùng bởi Role/Workflow KHÁC không?"
  3. Chỉ tách thành Skill khi: **(a)** tái sử dụng ≥ 2 nơi VÀ **(b)** đủ phức tạp (>30 dòng instruction)
  4. Giữ inline các khối nhỏ, đặc thù — tránh over-engineering
- **Anti-pattern:** Tách mọi thứ thành skill → Token overhead vì mỗi SKILL.md có ~500 tokens metadata + boilerplate

### B. Static Lookup Pattern (Thay thế Dynamic Loading)
- **Vấn đề:** Workflow cần tra cứu thư mục lưu trữ, nhưng đọc file index (`000-Index.md`) mỗi lần chạy tốn ~1,000 tokens
- **Giải pháp:** Hardcode bảng ánh xạ trực tiếp vào workflow instruction. Chỉ cần cập nhật khi cấu trúc thư mục thay đổi (hiếm khi)
- **Kết quả:** Tiết kiệm ~1,000 tokens/lần chạy, zero latency cho bước directory mapping

### C. Adaptive Confirmation Pattern
- **Vấn đề:** Quá nhiều bước xin confirm → User frustration, token bloat
- **Giải pháp:** Confidence-based — ≥85% thì gộp output + confirm; <85% thì hỏi trước
- **Kết quả:** Giảm 1-2 lượt tương tác cho ~80% trường hợp (high confidence)

## ⚠️ 3. Bẫy sai lầm & Cách tránh (Pitfalls & Prevention)

### Pitfall 1: Quên đối soát Documents-Template
- **Lỗi đã gặp:** Bug report template thiếu YAML frontmatter, lưu sai thư mục (`docs/035-QA/` thay vì `docs/035-QA/Reports/`), naming convention sai format
- **Nguyên nhân:** Viết workflow từ trí nhớ thay vì verify với `Documents-Template.md`
- **Cách phòng ngừa:** **BẮT BUỘC** đọc `knowledge-base/99-Templates/Documents-Template.md` trước khi viết bất kỳ workflow nào tạo file trong `docs/`. Đặc biệt kiểm tra: (1) đường dẫn sub-directory, (2) naming convention, (3) frontmatter fields

### Pitfall 2: Thiếu Edge Case cho Input rỗng
- **Lỗi đã gặp:** Workflow không xử lý khi User gõ `/opsx-ingest` không kèm URL
- **Nguyên nhân:** Tập trung vào happy path
- **Cách phòng ngừa:** Mọi workflow đều cần **Bước 0: Validate Input** — kiểm tra input tồn tại và hợp lệ trước khi bắt đầu xử lý

### Pitfall 3: Quên bước hậu kỳ (Post-creation)
- **Lỗi đã gặp:** Tạo file xong nhưng không cập nhật MOC file (vi phạm Documents-Template)
- **Cách phòng ngừa:** Khi workflow tạo file trong `docs/`, luôn thêm bước cuối: "Cập nhật MOC + Index nếu cần"

## 🎯 4. Ưu tiên của người dùng (User Preferences)
- **Token efficiency là ưu tiên hàng đầu**: User luôn hỏi "có cách tối ưu hơn không?" — cần chủ động đề xuất surgical fetching, static lookup, và adaptive confirmation
- **Thiết kế trước, implement sau**: User luôn dùng `/opsx-explore` trước khi code — tôn trọng quy trình Explore → Design → Implement
- **Self-review trước khi deliver**: User expect workflow/skill phải được tự review và sửa lỗi trước khi trình bày kết quả cuối cùng
